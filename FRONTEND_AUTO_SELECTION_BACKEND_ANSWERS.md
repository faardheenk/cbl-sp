# Backend Answers To Frontend Auto-Selection Caveats

## Context

This answers the six caveats raised in `FRONTEND_AUTO_SELECTION_BACKEND_CAVEATS.md`.

After reviewing the full backend pipeline — matching passes, match history, output handler, and how the combined output is generated — here are the confirmed answers and the specific backend changes needed.

## Caveat 1: Is `group_id` A Safe UI Group Identifier In Every Matched Bucket?

**Answer: Yes, with one gap we will fix.**

Current `group_id` assignment:

| Source | Format | When assigned |
|---|---|---|
| Pass 3 — corporate root | `NAME_GROUP_N_ROOT` | Multiple CBL rows share insurer indices via root match |
| Pass 3 — fuzzy clustering | `NAME_GROUP_N_FUZZY` | Multiple CBL rows grouped by fuzzy name similarity |
| Pass 3 — secondary root | `NAME_GROUP_N_SECONDARY` | CBL matched via secondary corporate root |
| Pass 3 — label match | `NAME_GROUP_N_LBL` | Individual label-based match |
| Pass 3 — merge overlapping | `MERGED_GROUP_N` | Groups merged when they share insurer indices |
| Match history — move | `HISTORY_N` | History pre-placed multi-CBL group |
| Match history — regroup | `HISTORY_REGROUP_N` | History regroup merged group |

Each `group_id` is generated by a per-run counter. No two unrelated groups share the same `group_id`. Rows only share `group_id` when the backend explicitly determined they belong together.

**The gap:** Pass 1 individual 1:1 exact matches do NOT get a `group_id`. These are single CBL rows matched to one or more insurer rows without any grouping. They account for the majority of exact matches.

For `group_id` to be the universal selection key, we need to assign it to individual matches too. The backend will assign a unique `group_id` to every matched CBL row that doesn't already have one (e.g. `MATCH_N`).

**Contract confirmed:**

```
Rows share group_id only when they should be selected/highlighted together.
Unrelated matches never share group_id.
Every matched row will have a group_id.
```

## Caveat 2: Do Insurer Rows Always Have The Same Group ID As Their Related CBL Rows?

**Answer: Not today. This is the core backend change.**

Currently, insurer rows in the source DataFrame (`insurer_df`) have no `group_id` column. The relationship is only expressed through CBL-side `matched_insurer_indices`.

The combined output (written by `explode_and_merge`) zips CBL and insurer columns onto the same row. The output already contains `group_id` on rows that have CBL data, but blank/spacer rows (where CBL side is empty) have `None` for `group_id`.

**What the backend will change:**

The output handler will propagate `group_id` to every row in a group, including insurer-only spacer rows. After `explode_and_merge` creates the combined output, every row in a group will carry the same `group_id` value regardless of which side has data.

This means:

```
Row  | ClientName | group_id  | ... | ClientName_INSURER | ...
0    | Client A   | G_001     | ... | Insurer X          | ...
1    |            | G_001     | ... | Insurer Y          | ...
2    | Client B   | G_001     | ... |                    | ...
```

Row 1 (blank CBL, has insurer) now carries `group_id = G_001` instead of `None`.

**Contract confirmed:**

```
Every non-blank row in a matched group carries the same group_id.
Blank spacer rows in the group also carry the same group_id.
No-match rows have empty/null group_id.
```

## Caveat 3: Do We Still Need Per-CBL Insurer Ownership?

**Answer: No, for selection and highlighting. Group-level is sufficient.**

Looking at the output handler:

- `_process_group_match` (`output_handler.py:64-96`) zips CBL and insurer rows positionally. Row position already provides visual per-CBL ownership in the displayed table.
- `_process_individual_match` (`output_handler.py:99-168`) shows CBL data only on the first row; subsequent insurer rows have blank CBL columns.

The visual layout already tells the user which insurer rows sit next to which CBL row. Selection/highlighting does not need to enforce this — clicking any row in the group should select the full group.

For regroup, the user explicitly selects specific rows. The frontend does not need per-CBL ownership metadata to know what the user selected — it knows from the selection state itself.

`matched_insurer_indices` remains available on CBL rows for any edge case, but the frontend should not need it for selection or highlighting.

**Contract confirmed:**

```
group_id is the unit of selection and highlighting.
Clicking any row in a group selects/highlights the full group.
Per-CBL ownership is a visual layout concern, handled by row position in the output.
```

## Caveat 4: Does Backend Preserve Parallel Layout And Blank Spacer Rows?

**Answer: Yes. No change to layout.**

The output handler already does this:

```python
# output_handler.py _process_group_match (line 83-94)
max_rows = max(len(group_cbl_rows), len(insurer_rows))
for i in range(max_rows):
    cbl_row = group_cbl_rows[i] if i < len(group_cbl_rows) else None
    insurer_row = insurer_rows[i] if i < len(insurer_rows) else None
    combined_row = _create_zipped_row(cbl_row, insurer_row, cbl_cols, insurer_cols)
```

When one side has fewer rows, the shorter side gets `None` values (blank spacer). This behavior does not change.

**Contract confirmed:**

```
Matched bucket sheets remain positionally aligned.
Blank spacer rows are emitted on the shorter side.
This behavior is unchanged by the group_id proposal.
```

## Caveat 5: What Happens To `matched_insurer_indices`?

**Answer: It stays. Fully populated. Unchanged.**

`matched_insurer_indices` is used internally by:

1. The output handler to fetch insurer rows for zipping (`_extract_insurer_indices`, `_get_insurer_rows_for_group`).
2. The orchestrator to compute insurer match statistics and determine unmatched insurer rows.
3. Pass 3 for group merging and regrouping decisions.

None of this changes. `matched_insurer_indices` continues to be set by:

- All matching passes (pass1, pass2, pass3)
- Match history move replay
- Match history regroup replay

**Contract confirmed:**

```
matched_insurer_indices remains populated consistently across all code paths.
group_id is the preferred frontend selection field.
matched_insurer_indices is available as a fallback if ever needed.
```

## Caveat 6: How Should No-Match Rows Be Marked?

**Answer: Empty/null `group_id`. Guaranteed.**

No-match rows never receive a `group_id` in any code path:

- The matching passes only assign `group_id` to matched groups.
- Match history no-match placement sets `match_status` but never sets `group_id`.
- Unmatched insurer rows are collected by subtracting matched indices from all indices — they have no CBL association and no `group_id`.

**Contract confirmed:**

```
No Matches CBL rows have null/empty group_id.
No Matches Insurer rows have null/empty group_id.
The frontend should not auto-select unrelated no-match rows together.
```

## Summary Of Backend Changes Needed

Two changes, both small:

### 1. Assign `group_id` to all matched rows (including individual 1:1 matches)

Currently, individual 1:1 matches from pass1 have no `group_id`. We will add a post-processing step after all passes that assigns a unique `group_id` (e.g. `MATCH_N`) to every matched CBL row that doesn't already have one.

Location: `orchestrator.py`, after all passes complete, before `explode_and_merge`.

### 2. Propagate `group_id` to all rows in the combined output

Currently, blank/spacer rows in the output have `None` for `group_id`. We will propagate the group's `group_id` to every row in the zipped output block, including insurer-only spacer rows.

Location: `output_handler.py`, in `_process_group_match` and `_process_individual_match`.

### What does NOT change

- `matched_insurer_indices` — stays fully populated
- Output sheet structure — same sheets, same layout
- Blank spacer row equalization — same behavior
- Fingerprints, history format, all existing fields — unchanged
- Matching pass logic — unchanged

## Next Step

Once the frontend confirms this answers all caveats, the backend will implement the two changes above.
